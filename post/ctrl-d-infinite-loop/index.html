Ctrl D 导致死循环？

典型的 REPL 程序， while 循环中读取命令行内容 while (scanf("%s", buf))，

  • Ctrl C 触发中断（SIGINT），程序退出
  • Ctrl D 在终端中表示输入结束，发送 EOF 信号（-1），程序死循环

就是说，在后一种情况下，等价于 while (-1)。修改也很容易 while(scanf("%s", buf) == 1)，异常情况下，while (-1 == 1) 会让循环正常终止。

问题变成了，为什么会等价于 while(-1)？在两个函数中分别调用 scanf，按顺序第一个函数触发 EOF，

void func1()
{
    int x = 11;
    scanf("%d", x); /* ctrl d */
    printf("x = %d\n", x);
}

void func2() { /* same as func1 */ }
...

结果 func2 的 scanf 没有等待用户输入。可以猜测，scanf 函数的实现中，使用了 static 变量作为判断标志，该函数不可重入，是非线程安全的（一般也不会想到在多线程中使用）。

进 glibc 看一眼， scanf 内部调用的 _IO_vfscanf_internal， 如果读取到了 EOF， 会触发 input_error(), 内部使用了 static 变量存储状态。glibc 的代码真晦涩，不想再看到了:)。

getchar() 也有同样的问题。
